  HackerOne #3601323 — pull_request_target credential theft

Use OBS Studio (free) or any screen recorder.  
Record at 1080p, speak slowly, narrate each step.

────────────────────────────────────────────────────
SCENE 1  — Show the Real Vulnerability  (≈2 min)
────────────────────────────────────────────────────

1. Open browser → navigate to:
   https://github.com/kubernetes/minikube/blob/HEAD/.github/workflows/functional_extra.yml

2. HIGHLIGHT & NARRATE each dangerous block:

   a) "This workflow triggers on pull_request_target with the labeled event.
       pull_request_target runs with WRITE permissions and access to secrets,
       even for fork PRs."

   b) SCROLL to the checkout step:
        - uses: actions/checkout@v6
          with:
            ref: ${{ github.event.pull_request.head.sha || github.ref }}

      "Here the workflow checks out the pull request HEAD — that's the
       FORK's code, not the upstream repo. This is the root cause."

   c) SCROLL to Azure login:
        - name: Azure Login
          uses: azure/login@v2
          with:
            creds: '{"clientId":"${{ secrets.MINIKUBE_AZ_CLIENT_ID }}", ...}'

      "Immediately after checking out untrusted code, the workflow
       authenticates to Azure with a full service principal — client ID,
       secret, subscription ID, and tenant ID all injected."

   d) SCROLL to the make commands:
        make e2e-windows-amd64.exe
        make minikube-windows-amd64.exe

      "The fork's Makefile now executes in an environment where all four
       Azure credentials are available. A single 'curl' in the Makefile
       exfiltrates them."

   e) "The only gate is the 'ok-to-extra-test' label, which must be
       applied by an OWNERS member. GitHub Security Lab explicitly
       states this is insufficient — a single social-engineering
       mistake exposes the entire Azure subscription."

11. SWITCH back to Account A (repo owner).

12. On the PR, apply the label:  ok-to-test

    "This simulates a maintainer approving the PR for CI. In minikube
     the label is 'ok-to-extra-test'."

13. Go to the Actions tab → SHOW the workflow running.

14. Open the workflow logs:

────────────────────────────────────────────────────
SCENE 3  — Show the Fix  (≈1 min)
────────────────────────────────────────────────────

16. Open .github/workflows/fixed-workflow.yml:
    "The fix is to use workflow_run. The privileged workflow only
     triggers after a clean validation workflow succeeds, and it
     checks out upstream code — never the fork's. Secrets are never
     exposed to attacker-controlled code."

────────────────────────────────────────────────────
SCENE 4  — Wrap Up  (≈30 sec)
────────────────────────────────────────────────────

17. "To summarize: the pull_request_target trigger combined with a fork
     checkout and secret injection allows any fork to steal Azure
     credentials when a maintainer applies a label. The label gate
     is a social-engineering target, not a security boundary.
     The recommended fix is workflow_run separation."
